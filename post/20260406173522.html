前言

 2026.4.6，经过半天的折腾和Gemini的鼎力支持，把STM32N6的IMX335例程搞定了，现在开始记录一下开发过程。这部分较为复杂，原因在于以下几点：

  1. 正点原子的原先例程中加入了其他非常多的无关模块，比如SD卡，FATFS文件系统，蜂鸣器，图片解析模块等，需要将IMX335的相关代码从中进行剥离。

  2. 部分例程中的代码没有经过cubemx生成，是后期人为添加的，给排查增添了不少的压力。

  3. IMX335本身的代码情况复杂，虽然有官方GitHub上的文件支持，但是从有和学会的跨度仍然较大。

  4. IMX335的图像显示涉及到了非常多摄像领域的知识，比如白平衡，曝光等等，整个例程的实现都是基于正点原子本身已经配置好的参数进行的，后期如果需要调整可能还需要去学习官方软件：STM32 ISP IQTune去专门设计画面。
     好，接下来就一步一步去拆解如何复现这个工程。

正文

1.软件和硬件准备

​ 硬件:STM32N6开发板，ST-LINK，电源线，数据线(用于串口输出提示)，正点原子通用RGB屏幕(博主的具体型号:LCD主控:0x4384 型号代码ATK-MD0430R-800480，触摸主控型号:GT911，I2C通信)，IMX335摄像头模块。
​ 软件:STM32CubeIDE，STM32CubeMX，RGB屏幕驱动库(正点原子官方,触摸没有用到)，串口驱动库，IMX335驱动库，中间件STM32_MW_ISP(图像算法，正点原子例程中有，若需要自行移植请移步GITHUB)。

2.原理解析（IMX335在STM32上如何工作）

2.1 接口协议解析

 这里我专门做了一个图来理清新协议之间的关系，如下:

​

可以看到，IMX335本身靠I2C与STM32通信，但是图像数据走的是速度更高的MIPI CSI协议，因此你可以在原理图中看到两种不同的协议引脚定义，并且，模块传过来的数据并不是最终我们在屏幕上看到的结果，而是摄影界常用的一个词”RAW”图像(原始图像格式),我们还需要通过主控中的DCMIPP技术对图像数据进行转换和裁剪(原始的IMX335图像一帧数据大小为2592x1944),最后将这部分数据搬到RGB_LCD显示的内存上,才有了我们看到的实时图像。

2.2 中间层 STM32_MW_ISP

 就像数码相机，手机镜头有专业模式一样，ST公司也为摄像头设计了一个类似的“专业模式”，通过把DCMIPP技术对图像数据进行转换和裁剪后的数据进行诸如白平衡，饱和度，色差等标准校准实现更优质的显示效果。但是这个专业模式需要我们自己将中间层加入工程中(cubemx中并没有包含，只能靠去GitHub下或者去拿正点原子现成的)，然后配合ST官方软件STM32 ISP IQTune去调节这些参数使得画面变得正常。

 好消息是正点原子已经调出了一版正常的参数在例程之中，我们本期不需要费老大劲去调这个东西，以后如果有需要这个调配过程可能需要单开一期讲解。

2.3 最终的初始化流程

 全局初始化时间线 (Boot Sequence)

 为了让上述管线正常运转，在你的 main.c 中，初始化的先后顺序有着严格的逻辑依赖：

  1. **底层通电 (HAL_Init, SystemClock_Config)**：先给整个芯片供电，把 PLL 时钟拉满。
  2. **引脚与内存就绪 (MX_GPIO_Init)**：配置好摄像头的复位引脚，并初始化外部 SRAM/HyperRAM，确保显存有物理载体。
  3. 显示端先就绪 (rgblcd_init())：必须先初始化屏幕，配置好 LTDC，让屏幕亮起背光并开始扫描显存（此时显存是空的，初始化后为全白）。
  4. **采集端初始化 (imx335_init())**：设置I2C 配置和 DCMIPP 外设配置(2592x1944 的分辨率、10-bit RAW 格式、30帧的帧率)，同时在内部调用 ISP_Init，将 ISP 算法与这套管线绑定。
  5. **中断与底层总线就绪 (HAL_DCMIPP_MspInit)**：打开 DCMIPP 和 CSI 的时钟，并开启了 NVIC 全局中断。
  6. **打开水龙头 (imx335_start_capture(...))**：启动硬件 DMA，数据正式开始从传感器流向RGB_LCD显存。
  7. **进入主循环 (while(1))**：CPU 此时已经不需要管图像搬运了(DMA自动接管)，它在循环里不断执行 imx335_isp_background_process()，利用帧中断传来的统计数据，默默地计算白平衡和曝光补偿，再通过 I2C 悄悄调整摄像头的寄存器，维持画面色彩的完美。

4.4.2 撰写回调函数

 在cubemx中，我们打开了关于DCMIPP的两个中断，现在我们需要将对应的回调函数补上，如下：

//main.c文件下
/* USER CODE BEGIN 4 */
/**
 * @brief   HAL库DCMIPP pipe帧事件回调函数
 * @param   pHdcmipp: DCMIPP句柄指针
 * @param   Pipe: DCMIPP pipe号
 * @retval  无
 */
void HAL_DCMIPP_PIPE_FrameEventCallback(DCMIPP_HandleTypeDef *hdcmipp, uint32_t Pipe)
{
  imx335_dcmipp_pipe_frame_cb(hdcmipp, Pipe);
}

/**
 * @brief   HAL库DCMIPP pipe Vsync事件回调函数
 * @param   pHdcmipp: DCMIPP句柄指针
 * @param   Pipe: DCMIPP pipe号
 * @retval  无
 */
void HAL_DCMIPP_PIPE_VsyncEventCallback(DCMIPP_HandleTypeDef *hdcmipp, uint32_t Pipe)
{
  imx335_dcmipp_pipe_vsync_cb(hdcmipp, Pipe);
}
/* USER CODE END 4 */

4.5 修改引用路径

 这一次我们使用了非常多不同类型的文件，文件结构复杂，因此原先基础部分的文件引用结构不够用了，需要添加以下新规则：

/${ProjName}/Drivers/BSP
/${ProjName}/Middlewares/STM32_MW_ISP/evision/Inc
/${ProjName}/Middlewares/STM32_MW_ISP/isp/Inc
/${ProjName}/Application/User/STM32_MW_ISP

4.6 添加库文件

 STM32_MW_ISP和DSP库一样，都需要使用封装好的lib库，因此我们需要添加，具体路径如下：

  1. 在 CubeIDE 左侧资源管理器右键点击你的工程名，选择 Properties (属性)。
  2. 展开 C/C++ Build -> 选择 Settings。
  3. 在中间偏右的选项卡中，找到 Tool Settings。
  4. 在下面的树状菜单里，展开 MCU GCC Linker -> 点击 Libraries。
  5. 在右上角的 Libraries (-l) 窗口中点击带有绿色加号 + 的图标添加库。
  6. 添加两条库名:n6-evision-awb_gcc 和 n6-evision-st-ae_gcc。
  7. 在右下角的 Library search path (-L) 窗口中：点击带有绿色加号 + 的图标添加路径。
  8. 填入”${workspace_loc:/${ProjName}/Middlewares/STM32_MW_ISP/evision/Lib}”。
  9. 点击 Apply and Close。

4.7 编译运行

 经过以上步骤，我们完成了所有的流程，现在开始编译调试和运行。
 这里有一个BUG需要提醒，就是在开发板不断电运行DEBUG程序的情况下，再次烧录重启时（热启动），画面只在左上角显示一小块；但是如果在烧录前断电再重新上电，画面就显示正常，这是因为热启动时残留的图像电平干扰了屏幕 ID 读取机制，导致其退化回 480x272 安全分辨率。
 下面是一些遇到的BUG及其解决方法：

  1. 画面诡异的“绿色滤镜”（Bayer 阵列原色）
    问题：画面能动，但全屏发绿。
    解决：这是因为 ISP 的 3A 算法未收到统计数据。发现 main.c 底部遗漏了中断回调函数的实现。补充了 HAL_DCMIPP_PIPE_FrameEventCallback 和 HAL_DCMIPP_PIPE_VsyncEventCallback，将硬件中断成功桥接到 imx335.c，唤醒白平衡算法。

  2. 画面不刷新（仅显示 “IMX335 OK!”）
    问题：程序正常进入 while(1)，但屏幕白屏，DCMIPP 数据没有流向显存。
    解决：在 CubeMX 中补全了 IC17、IC18 时钟树路由，开启了 DCMIPP 和 CSI 时钟，执行了硬件复位，并开启了 DCMIPP_IRQn 和 CSI_IRQn 全局中断。

  3. I2C 通信失败（卡死在 ReadID，仅显示 “IMX335 ERROR!”）
    问题：代码成功烧录，但卡死在摄像头初始化阶段的 IMX335_ReadID。
    解决：摄像头处于“断电/死机”状态。在 CUBEMX中补充对 PG6 (PWDN) 和 PG4 (RST) 引脚的推挽输出初始化。

5. 最终结果

 运行后应该可以在屏幕上看到完整的摄像样貌，颜色正常，由于摄像头素质拍出来的可能比较糊但是能辨认物品。
