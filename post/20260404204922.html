前言

 2026.4.4，经过半天多的折腾，终于把STM32N6的RGB_LCD例程搞定了，现在开始记录一下开发过程。相比于H7，N6的开发过程异常艰难，有些地方非常不一样。

正文

1. 硬件准备

 STM32N6开发板，ST-LINK，电源线，数据线(用于串口输出提示)，正点原子通用RGB屏幕(博主的具体型号:LCD主控:0x4384 型号代码ATK-MD0430R-800480)。

2. 软件准备

 STM32CubeIDE，STM32CubeMX，RGB屏幕驱动库(正点原子官方)，串口驱动库。

3. 环境搭建

配置RIF(新机制，重要)

 资源隔离框架（RIF）,用于隔离不同安全等级的代码和数据，防止低安全等级的代码访问高安全等级的代码和数据。这个机制会影响芯片与外设的通信，因此一定要打开权限，否则你的代码会无法与外设通信。

3.2 STM32CubeIDE

 打开STM32CubeIDE，导入工程，这里不再赘述,用到的BSP板级支持包有：UART,HyperRam,RGBLCD,SYS,请将他们也一并引入工程这里不再赘述。
 接下来就是修改代码了。

修改链接脚本

 这一步非常坑，上一期中我们提到，“这部分内容由于正点原子已经有完整的教程，因此这里不再赘述，直接给跳转链接”，但是,事实是，正点原子的教程中并没有给出正确的STM32N647X0HXQ_LRUN_RAMxspi1.ld链接脚本，导致RGBLCD的部分代码(最典型的就是字符库)没有地方放，进而导致整个程序无法被烧录，一直报错:

Error in final launch sequence:
Failed to execute MI command:
load D:\\04_RGB_LCD\\STM32CubeIDE\\Appli\\Debug\\04_RGB_LCD_Appli.elf
Error message from debugger back end:
Load failed
Failed to execute MI command:
load D:\\04_RGB_LCD\\STM32CubeIDE\\Appli\\Debug\\04_RGB_LCD_Appli.elf

 仔细观察链接脚本,你会发现它的教程只更改了两个内容:MEMORY和给SECTIONS添加了一个EXTRAM。但是事实是，在SECTION中，还有很多内容指向的ROM,而之前按教程修改的MEMORY中已经没有了ROM，因此这些内容没有地方放，导致程序无法被烧录：

/* Memories definition */
MEMORY
{
RAM    (xrw)    : ORIGIN = 0x34000400,   LENGTH = 2047K
EXTRAM (rw)     : ORIGIN = 0x90000000,   LENGTH = 32M
}

/* Sections */
SECTIONS
{
/* External RAM */
.EXTRAM (NOLOAD):
{
    . = ALIGN(8);
    *(.EXTRAM)
    . = ALIGN(8);
} >EXTRAM
...
/* The program code and other data into "ROM" Rom type memory */
.text :
{
    . = ALIGN(4);
    *(.text)           /* .text sections (code) */
    *(.text*)          /* .text* sections (code) */
    *(.glue_7)         /* glue arm to thumb code */
    *(.glue_7t)        /* glue thumb to arm code */
    *(.eh_frame)
    *(.RamFunc)        /* .RamFunc sections */
    *(.RamFunc*)       /* .RamFunc* sections */

    KEEP (*(.init))
    KEEP (*(.fini))

    . = ALIGN(4);
    _etext = .;        /* define a global symbols at end of code */
} >ROM

1. 将所有段落的目标存储器从 >ROM 改为 >RAM
before: 
} >ROM

after: 
} >RAM

2. 取消 .data 段的加载地址分离（LMA）
before:
} >RAM AT> ROM

after:
} >RAM

3. 添加 GCC 11+ 的 (READONLY) 属性,优化编译器
before:
.init_array :
after:
.init_array   (READONLY) : /* The READONLY keyword is only supported in GCC11 and later, remove it if using GCC10 or earlier. */

 好，接下来讲讲为什么需要这么做。
 首先，在main.c中，我们看到了很多ifdef DEBUG的预处理器指令，和以往的F1,F4,H7的工程都不太一样。原因还是因为STM32N6新架构的问题，在初探那期，我们知道了一个完整的STM32N6工程实际上包含了两个部分(两个工程)：FSBL和Application,FSBL由于正点原子已经给我们写了一个通用的，因此我们只需要关注Application部分。这几期实战就是这样。
 那么,FSBL里面存了什么呢？我们来看看:

//FSBL工程main.c
/**
    * @brief  The application entry point.
    * @retval int
    */
int main(void)
{
    SCB_EnableICache();

    SCB_EnableDCache();

    HAL_Init();

    /* Configure the system clock */
    SystemClock_Config();//看这里！！！！

    /* Initialize all configured peripherals */
    MX_GPIO_Init();
    MX_XSPI1_Init();
    MX_XSPI2_Init();
    MX_EXTMEM_MANAGER_Init();
    /* USER CODE BEGIN 2 */
    HyperRAM_Init(&HyperRAMObject, &hxspi1);         //看这里！！！！
    HyperRAM_EnableMemoryMappedMode(&HyperRAMObject);//看这里！！！！

    /* USER CODE END 2 */

    /* Launch the application */
    if (BOOT_OK != BOOT_Application())
    {
    Error_Handler();
    }
    /* Infinite loop */
    while (1)
    {
    
    }
}

在其中你可以看到一个SystemClock_Config()函数，以及我们刚刚在RGB工程中在ifdef DEBUG的预处理器指令中看到的HyperRAM_Init(&HyperRAMObject, &hxspi1);和HyperRAM_EnableMemoryMappedMode(&HyperRAMObject);。
 SystemClock_Config()函数是老熟人了，F1,F4,H7的工程中都有。
 HyperRAM_Init(&HyperRAMObject, &hxspi1);和HyperRAM_EnableMemoryMappedMode(&HyperRAMObject);这两个函数用于将外部SRAM映射到内存之中，使得外部SRAM可以被访问。
 那么，我们就彻底搞清了这一切的来龙去脉。
 在我们编写的Application工程中,没有SystemClock_Config()函数,你用全局搜索也找不到，因为在FSBL中已经定义了。但是，在开发阶段，也就是开发者模式下程序跑在内存上时，我们没有初始化我们在时钟树中所配置的内容！！！，你可以用以下代码验证一下没有了那些ifdef DEBUG的预处理器指令时(屏蔽sys_clock_config_debug();)，你的程序运行在什么频率上：

printf("\r\n================ System Clock Info ================\r\n");

 可以看到差距显著，所以我们必须要使用这些ifdef DEBUG之间的代码去填补我们写在FSBL中的内容，以此保证程序正常运行。

 而对于HyperRAM_Init(&HyperRAMObject, &hxspi1);和HyperRAM_EnableMemoryMappedMode(&HyperRAMObject);,这两个函数加入的原因实际上是因为RGBLCD代码中的这句话

/* LTDC帧缓冲区 */
uint16_t g_ltdc_lcd_framebuf[1280 * 800] __attribute__((section(".EXTRAM")));

​ 他使用了外部SRAM作为存储LTDC帧的硬件，因此必须要在开始将外部内存初始化让主控得以正常寻址到外部内存之中。

 经过这部分设置之后，就可以按照之前实战1的步骤进行调试验证和检查了，如果正常运行的话，你应该可以看到屏幕的背景是不停切换的。并且文字内容正常显示。

总结

​ 通过以上步骤，我们成功在STM32N6上添加了RGBLCD模块，并实现了显示功能。在后续的开发中，我们可以根据需要添加出没和一些显示组件，比如LVGL以实现更多的功能。后续文章将会继续介绍STM32N6的更多功能。
